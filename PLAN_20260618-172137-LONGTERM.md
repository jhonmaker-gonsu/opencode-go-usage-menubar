# Plan: Long-term correctness verification of opencode.5m.sh

## Goal

Prove that `~/SwiftBar/opencode.5m.sh` (v1.2.0) keeps updating correctly
**over time** — across day/week/month boundaries, opencode version
upgrades, schema drift, CLI breakages, and edge-case data states — not
just at the moment of initial verification. Classify each plausible
failure mode by likelihood/impact, design mechanical or synthetic
tests for each, identify bugs the current script would NOT survive,
and propose fixes.

Scope: the script is read-only — no edits to `/Users/gon/SwiftBar/opencode.5m.sh`
or to `PLAN_20260618-172137.md`. The script's env-var overrides
(`BUN`, `JQ`, `NUMFMT`, `SQLITE`, `OC_DB`) are the test hooks.

## Bugs found in current script (v1.2.0)

### Bug 1 — Silent zero on schema/filter drift (CRITICAL)

- **Symptom**: SQL returns 0% when (a) `data.$.cost` is renamed, (b)
  `data.$.providerID` value changes, (c) `data` is not JSON.
- **Root cause**: `COALESCE(SUM(json_extract(data,'$.cost')), 0)` makes
  any extraction failure look like 0. The script then happily prints
  `S: 0% W: 0% | color=#33dd33` — the user sees green and assumes
  "all good".
- **Affected modes**: 6, 7, 17, plus B3/B4/B5/B19.
- **Severity**: **critical** — user would burn through quota thinking
  they had plenty left. This is exactly the long-term-correctness
  failure the user asked us to rule out.

### Bug 2 — jq "no match" returns 0 (CRITICAL)

- **Symptom**: When `opencode-usage` JSON schema changes (e.g.,
  `periods` → `days`, `totals.input` → `totals.tokens_in`), the jq
  selector finds nothing, returns 0 with empty stdout. The script's
  `if ! ROW=$(...)` evaluates false, `JQ_RC` stays 0, and execution
  proceeds to the `if [[ -z "$ROW" ]]` branch — which shows
  `S: <sql-pct>% W: <sql-pct>%` with `Tokens in: 0`. The user sees
  valid-looking percentages and "0 tokens today". They may not notice
  for hours.
- **Root cause**: jq's "selector matched nothing" is a successful run,
  not an error. `JQ_RC` capture (introduced in v1 plan) only catches
  jq syntax errors.
- **Affected modes**: 9, 16, plus B6/B7.
- **Severity**: **critical** — same outcome as bug 1 from the user's
  perspective.

### Bug 3 — TZ mismatch with $TODAY (MEDIUM)

- **Symptom**: For users west of UTC, after ~16:00 local the CLI's
  `.date` (UTC) has already rolled to the next day, but `$TODAY`
  (local) is still "yesterday". jq finds no match → "Tokens in: 0"
  in the dropdown. For users east of UTC, the inverse happens in
  the morning.
- **Root cause**: v1.2.0 still has `TODAY=$(date +%Y-%m-%d)` (local)
  with no `TZ=UTC` fix. R17 in the original plan flagged this; it
  was never implemented.
- **Affected modes**: 1, 5, 18.
- **Severity**: **medium** — the SQL %s remain correct (UTC-anchored
  windows), so quota visibility is preserved; only the "today's
  tokens" line in the dropdown glitches at the TZ boundary.

### Bug 4 — Stale data UX gap (LOW)

- **Symptom**: After opencode session ends, costs stay flat. The
  menubar shows a frozen %. No "data from HH:MM" or "stale" indicator.
- **Severity**: **low** — cosmetic, but it would be the first thing a
  long-term user notices when they stop using opencode for a week.

## Proposed fixes

### Fix 1 — Cross-check SQL cost against CLI cost (bug 1)

After the SQL and CLI paths both complete, compare the SQL-5h cost
to the CLI's "today" cost. If `COST_5H = 0` AND the CLI reported any
non-zero cost today, emit a warning line in the dropdown: "Schema
mismatch: SQL reported $0 but CLI reported $X.XX — opencode may
have changed." This makes bug 1 visible without false positives on
genuine quiet days (CLI=0 also → no warning).

```bash
if awk -v a="$COST_5H" -v b="$COST_RAW" 'BEGIN{exit !(a+0==0 && b+0>0)}'; then
  echo "WARNING: SQL/CLI cost mismatch (possible schema drift) | color=#ff3333"
fi
```

Place this before the final `Refresh` line in both the empty-ROW
and happy paths. Adds 3 lines.

### Fix 2 — Use `jq -e` and post-validate the result (bug 2)

Change the jq invocation to:
1. Use `jq -e` so a no-match result exits with code 1.
2. After the `IFS= read`, also assert that `$IN` matches a number
   pattern (`[[ "$IN" =~ ^[0-9]+(\.[0-9]+)?$ ]]`).
3. If the assertion fails, route to the existing `Malformed row`
   branch.

```bash
if ! ROW=$(echo "$JSON" | "$JQ" -re --arg t "$TODAY" '.periods[]? | select(.date == $t) | [.totals.input // 0, .totals.output // 0, .totals.cost // 0] | @tsv' 2>/dev/null); then
  JQ_RC=$?
  ROW=""
fi
```

`jq -e` exits 1 on `false`/null. For a no-match (empty stream), `-e`
combined with `--arg t "$TODAY"` will still exit 0 unless the result
is literally `false` or `null`; to be safe, add an explicit
post-check: if `ROW` is empty AND the input JSON had at least one
period, treat it as a schema mismatch.

```bash
PERIOD_COUNT=$(echo "$JSON" | "$JQ" '.periods | length' 2>/dev/null || echo 0)
if [[ -z "$ROW" && "$PERIOD_COUNT" -gt 0 ]]; then
  echo "S: ?% W: ?% | color=#ff8800"
  echo "---"
  echo "opencode-usage JSON has periods but no match for $TODAY — schema may have changed"
  ...
fi
```

### Fix 3 — Detect and align `$TODAY` to CLI's timezone (bug 3)

Probe the CLI's `.date` format once per run and align `TODAY` to it:

```bash
FIRST_DATE=$(echo "$JSON" | "$JQ" -r '.periods[0].date // empty' 2>/dev/null)
TODAY=$(date +%Y-%m-%d)
if [[ -n "$FIRST_DATE" ]]; then
  CLI_DAY=${FIRST_DATE:0:10}
  LOCAL_DAY=$TODAY
  if [[ "$CLI_DAY" != "$LOCAL_DAY" ]]; then
    # CLI's "today" differs from local "today" → use CLI's day boundary
    # by setting TODAY to CLI's date
    TODAY="$CLI_DAY"
  fi
fi
```

This is heuristic: if the CLI is in UTC and local is JST, on a JST
morning before 09:00 the CLI's first period is still "yesterday UTC"
and the local user wants "today JST". The fix above would set
TODAY=yesterday-UTC, which then matches no period (because the
period is yesterday's bucket, not today's). The CORRECT fix is to
set `TZ=UTC` for the jq match only, and keep `TODAY` for human
display:

```bash
TODAY=$(date +%Y-%m-%d)
TODAY_UTC=$(TZ=UTC date +%Y-%m-%d)
# Use $TODAY_UTC for jq match, $TODAY for display
ROW=$(echo "$JSON" | "$JQ" -r --arg t "$TODAY_UTC" '.periods[]? | select(.date == $t) | ...' 2>/dev/null)
```

Then label the dropdown "Date (local): $TODAY" and "Date (CLI): $TODAY_UTC"
so the user sees the offset.

### Fix 4 — Add "last refresh" indicator (bug 4)

Add `echo "Last refresh: $(date +%H:%M:%S)"` in the dropdown. Three
lines, no behavioral change, but a user who returns to the menubar
after 6 hours immediately sees "Last refresh: 11:42" and knows the
data is from 6 hours ago.

- **A `faketime`-style wrapper** for time-based tests (B-midnight, C1).
  macOS doesn't have `faketime` ported cleanly; `libfaketime` works
  on Linux. For B-midnight, we instead test the **components** the
  script uses: `date +%Y-%m-%d` at a controlled TZ, and the SQL
  window. The integration is small enough that a unit-level check
  is sufficient.
- **Mocking the SwiftBar runner**. SwiftBar is not installed on
  this machine. Per the original plan, the script is a plain bash
  script whose stdout SwiftBar renders — testing stdout is
  equivalent. The SwiftBar restart behavior (C5) is a
  production-observation item, not a unit test.
- **Adding `--validate` mode to the script itself**. Tempting
  (would let the script self-test), but the user said "do not modify
  the script" and we should respect that boundary. All tests are
  external.
- **Replacing SQL with `opencode-usage --provider`**. R5 of the
  original plan rejected this because `opencode-usage` may not
  expose 5h/7d rolling windows. Confirmed by the v1.2.0 CLI flag
  set (`--since` is the only window knob, and it's a calendar
  boundary). Keeping SQL for SQL-shaped windows; using CLI for
  today's totals is the right division of labor.
- **Schema-pinning via SHA256 of `opencode-usage --help` output**.
  Cute, but the CLI updates legitimately; we'd just be tracking
  upstream version drift. The real signal is the assertion
  failures in B6/B7, not a hash of `--help`.
- **Hash-based check of `data.$.cost` on every row**. O(n) cost
  per refresh; too slow for a 5m plugin. Cross-check against
  the CLI (Fix 1) is O(1).
- **A "schema version" sentinel row in the DB**. Requires the
  opencode app to write it. Out of scope for our script.
- **Banning jq entirely** in favor of `python3 -c` JSON parsing.
  Tempting because Python's `json` module is more strict, but
  the user's `~/.bun/bin/bun` is already on PATH; adding python3
  to the dep list expands the surface area for a small gain. The
  right answer is `jq -e` + post-validation (Fix 2).
- **Frequency-increase to 1m refresh to catch bugs faster**.
  Rejected by R in original plan; 5m is plenty for a status
  indicator and the bug-detection work is done by tests, not by
  more polling.
- **Watching opencode's release notes for breaking changes**.
  Useful for C6, but not automatable from inside this script.
  The verifier should at least subscribe to the opencode GitHub
  releases feed; for this plan, we just document the watch item.
