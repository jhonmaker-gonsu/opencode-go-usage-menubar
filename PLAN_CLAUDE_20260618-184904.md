# Plan: SwiftBar plugin for Claude Code usage

## Goal

A single executable SwiftBar plugin that surfaces **Claude Code subscription
message usage** in the macOS menubar, reading local `~/.claude/projects/`
JSONL files directly (no Anthropic API, no third-party CLI). The plugin
is the Funky Falcon replacement the user asked for: 5-minute menubar
readout of "messages used / cap" against the Pro/Max subscription limits.

Final deliverable:

- `~/SwiftBar/claude.5m.sh` — the plugin itself (~150-200 lines).
- This plan file (workdir scratch only).

Menubar line shape (target): `CC 12/45 27%` (messages used / tier cap /
% of 5h window). Dropdown shape (target): plan label, 5h / 7d / 30d
message counts with %, per-window token totals (in/out/cache), "last
activity" timestamp, refresh action, plan-cap legend. Menubar line
capped to 14 chars via `length=14` (same convention as opencode plugin).

## Constraints

- **Stake**: small-medium (single bash script ~150-200 lines; bigger than
  opencode because no external CLI to wrap — we own the JSONL parsing).
- **Output file is the SwiftBar plugin** — SwiftBar mandates specific
  header comments (`<xbar.title>`, `<xbar.dependencies>`). These are
  tool-required metadata, not AI commentary; the no-comments rule is
  relaxed for them.
- **No emojis** in code or plan (user said no).
- **MD for the plan file**; shell for the script.
- **Tooling already present** on this Mac: `jq` (`/opt/homebrew/bin/jq`),
  `numfmt` (`/opt/homebrew/bin/numfmt`), `bash` 5.2.37 (zsh env, but
  script uses bash shebang). SwiftBar is installed. No `bunx` needed
  (we read local files only).
- **Source of truth is local files**, not server. Claude Code's
  `~/.claude/projects/<project-slug>/<session-uuid>.jsonl` is the only
  authoritative client-side record. No remote API.
- **Metric is MESSAGES, not tokens or dollars.** Pro/Max caps are
  per-message (Pro=~45/5h, Max5x=~225/5h, Max20x=~900/5h). Tokens
  appear in dropdown only. This is the single biggest departure
  from the opencode plugin's cost-based design.
- **Subscription tier is unknown to the plan.** User is one of
  Pro / Max 5x / Max 20x (or higher). Default to Pro. Allow env var
  override `CLAUDE_PLAN=pro|max5x|max20x` (v1 supports 3 tiers).
- **All projects aggregated by default.** `~/.claude/projects/` has
  one subdir per project. Allow `CLAUDE_PROJECT=<slug>` for
  per-project mode.
- **Env is macOS** — bash 3.2.57 ERE does not interpret `\t`; use
  `IFS=$'\t' read` for TSV (carried over from opencode plan R3 lesson).
- **No comments** beyond the SwiftBar metadata header.
- **Multi-agent workflow** — planner (this file) → implementer →
  verifier. Verifier will run mechanical oracle commands; they are
  specified below.

## Approach (numbered, file:line targeted)

Target file: `~/SwiftBar/claude.5m.sh` (single file, ~150-200 lines).
Line numbers are approximate; the implementer may re-flow within a band.
References like `R#` point to the opencode plan's risk register; new

### Section 1: SwiftBar metadata + strict mode + tool paths (L1-L25)

1. **L1 — shebang**: `#!/usr/bin/env bash`.

2. **L2-L6 — SwiftBar metadata header** (required, not commentary):
   - `# <xbar.title>Claude Code Usage</xbar.title>`
   - `# <xbar.version>1.0.0</xbar.version>`
   - `# <xbar.author>gon</xbar.author>`
   - `# <xbar.desc>Claude Code message quota (5h + 7d + 30d) from local JSONL</xbar.desc>`
   - `# <xbar.dependencies>jq,numfmt</xbar.dependencies>`
   - No `<swiftbar.refreshOnOpen>` — 5m filename refresh is canonical
     (carried over from opencode R18).

3. **L8 — strict mode**: `set -euo pipefail` (opencode R2 lesson: we
   want non-zero exit on jq failure so the `?` fallback can take over).

4. **L10-L15 — tool paths with overrides for testability**:
   - `JQ="${JQ:-/opt/homebrew/bin/jq}"`
   - `NUMFMT="${NUMFMT:-/opt/homebrew/bin/numfmt}"`
   - `CLAUDE_PROJECTS="${CLAUDE_PROJECTS:-$HOME/.claude/projects}"`
   - `CLAUDE_PLAN="${CLAUDE_PLAN:-pro}"` (pro | max5x | max20x)
   - `CLAUDE_PROJECT="${CLAUDE_PROJECT:-}"` (empty = aggregate all)

5. **L17-L25 — tier → cap lookup table** (hardcoded, mirrors Anthropic
   public docs; v2 should source from a config line at the top):
   ```bash
   case "$CLAUDE_PLAN" in
     pro)    CAP_5H=45  ;;
     max5x)  CAP_5H=225 ;;
     max20x) CAP_5H=900 ;;
     *)      CAP_5H=45  ;;
   esac
   ```
   The 7d and 30d caps are NOT publicly documented per-message for
   Pro/Max (Anthropic markets as "unlimited" with throttling at 5h).
   The dropdown will show a "5h is the binding constraint" hint and
   treat 7d/30d as informational. Do not invent caps for 7d/30d.

### Section 2: Aggregate JSONL discovery + time-window filter (L27-L55)

6. **L27-L32 — projects-dir existence guard** (no-data path):
   ```bash
   if [[ ! -d "$CLAUDE_PROJECTS" ]]; then
     echo "CC: no data"
     echo "---"
     echo "Last refresh: $NOW_HMS"
     echo "Refresh | refresh=true"
     exit 0
   fi
   ```
   EC5 exercises this via `CLAUDE_PROJECTS=/nonexistent`.

7. **L34-L37 — jq existence guard** (new for Claude plugin; opencode
   had bunx but the same pattern applies — tool-missing must not
   silently miscompute):
   ```bash
   if [[ ! -x "$JQ" ]]; then
     echo "CC ?"
     echo "---"
     echo "jq not found at $JQ"
     echo "Install: brew install jq"
     echo "Refresh | refresh=true"
     exit 0
   fi
   ```
   EC6 exercises this.

8. **L39-L48 — JSONL file discovery** (skip subagents/ to avoid
   double-counting; subagent messages are spawned by the parent
   assistant turn and should not be billed as separate user messages):
   ```bash
   NOW_S=$(date +%s)
   CUTOFF_S=$(( NOW_S - 2592000 ))  # 30d, the widest window
   mapfile -d '' JSONL_FILES < <(
     find "$CLAUDE_PROJECTS" \
       -name "*.jsonl" \
       -not -path "*/subagents/*" \
       -mtime -31 \
       ${CLAUDE_PROJECT:+-path "*/$CLAUDE_PROJECT/*"} \
       -print0 2>/dev/null
   )
   ```
   - `-mtime -31` bounds the read to files touched in the last month
     (5m × 30d ≈ 8640 invocations × 524MB / 5m = bounded cost).
   - `${CLAUDE_PROJECT:+-path ...}` is a bash parameter expansion
     trick: only adds the `-path` filter when the var is non-empty
     (avoids running with no filter at all when CLAUDE_PROJECT="").
   - `mapfile -d ''` reads NUL-separated paths safely (handles
     project dirs with spaces — verified by `ls ~/SwiftBar/` which
     has none, but defensive).
   - In aggregate mode, 194 files / 524MB / ~10-30s jq parse per
     refresh. Acceptable for 5m cadence.

9. **L50-L55 — empty-file-set guard**:
   ```bash
   if [[ ${#JSONL_FILES[@]} -eq 0 ]]; then
     echo "CC 0"
     echo "---"
     echo "No JSONL sessions found in last 30d"
     echo "Refresh | refresh=true"
     exit 0
   fi
   ```
   This is a separate state from "no data" (the dir exists but
   nothing has been written). Both render to a different menubar
   line so the user can tell them apart.

### Section 3: Stream-parse JSONL with jq → TSV → awk window filter (L57-L95)

10. **L57-L70 — the core parse** (single pipeline; one jq process
    per file but piped through awk for windowing to avoid spawning
    N processes):
    ```bash
    TSV=$(
      for f in "${JSONL_FILES[@]}"; do
        "$JQ" -c 'select(.type == "assistant") | {ts: .timestamp, in: .message.usage.input_tokens, out: .message.usage.output_tokens, cache_create: .message.usage.cache_creation_input_tokens, cache_read: .message.usage.cache_read_input_tokens, model: .message.model}' < "$f" 2>/dev/null
      done \
      | "$JQ" -r '"\(.ts // "1970-01-01T00:00:00.000Z")|\(.in // 0)|\(.out // 0)|\(.cache_create // 0)|\(.cache_read // 0)|\(.model // "?")"' \
      | awk -F'|' -v now="$NOW_S" '
          {
            # ISO 8601 → epoch via gsub (date -d not portable)
            ts = $1
            gsub(/-|T|:|Z|\..*/, " ", ts)
            split(ts, t, " ")
            e = mktime(t[1]" "t[2]" "t[3]" "t[4]" "t[5]" "t[6])
            if (e == -1) e = 0
            age = now - e
            if (age >= 0 && age <= 18000)   c5++
            if (age >= 0 && age <= 604800)  c7++
            if (age >= 0 && age <= 2592000) c30++
            t5_in  += $2;  t5_out  += $3;  t5_cc  += $4;  t5_cr  += $5
            t7_in  += $2;  t7_out  += $3;  t7_cc  += $4;  t7_cr  += $5
            t30_in += $2;  t30_out += $3;  t30_cc += $4;  t30_cr += $5
            if (age >= 0 && age <= 18000 && $6 != "") models[$6]++
          }
          END {
            printf "%d\t%d\t%d\t%d\t%d\t%d\t%d\t%d\t%d\t%d\t%d\t%d\n", \
              c5, c7, c30, \
              t5_in, t5_out, t5_cc, t5_cr, \
              t7_in, t7_out, t7_cc, t7_cr, \
              t30_in, t30_out, t30_cc, t30_cr
            for (m in models) printf "M\t%s\t%d\n", m, models[m]
          }' 2>/dev/null
    ) || TSV=""
    ```
    - **Two-stage jq**: first `jq -c` filters `type=="assistant"` and
      shapes to the fields we need (cheap select). Second `jq -r`
      flattens to TSV with the field order awk expects. The
      intermediate object shape protects against `usage` being
      absent (every field defaults to 0 or "?").
    - **awk does windowing + aggregation**. Spawning one process per
      file would be N×; awk-once-on-the-stream is O(N) memory and
      1 process. Window check is strict `>= 0 && <= window` so the
      5h boundary is inclusive of "exactly 18000s ago" (opencode
      LONGTERM B9 lesson — be explicit about boundary semantics).
    - **mktime in awk** converts ISO 8601 → epoch without `date -d`
      (non-portable on macOS bash). `gsub(/-|T|:|Z|\..*/, " ", ts)`
      rewrites `2026-05-23T05:16:50.443Z` to `2026 05 23 05 16 50 `
      which mktime parses.
    - **Models dict** captures which Claude models were used in the
      5h window; surfaced in dropdown as "Models used: sonnet, opus".
    - The whole pipeline swallows stderr (`2>/dev/null` on each jq)
      but if jq fails on any file, the whole pipeline still produces
      partial output (the rest of the files parse normally). If
      jq fails on ALL files, `$TSV` is empty and we route to `CC ?`
      via the empty-row check below (L75).
    - **Performance**: 194 files × 50ms each = ~10s aggregate.
      Acceptable for 5m cadence. If user reports slowness, mitigation
      is to cache the parse to a tmpfile with mtime gate — deferred
      to v2 (analogous to opencode's deferred background-cache
      pattern).

11. **L72-L80 — parse TSV, guard empty / malformed** (opencode R3
    lesson: IFS= read NOT regex, restore IFS afterward):
    ```bash
    if [[ -z "$TSV" ]]; then
      echo "CC ?"
      echo "---"
      echo "jq parse produced no rows — possible schema drift"
      echo "Last refresh: $NOW_HMS"
      echo "Refresh | refresh=true"
      exit 0
    fi

    # First line = counts + tokens; subsequent lines = model usage
    FIRST_LINE=$(echo "$TSV" | head -1)
    OLD_IFS=$IFS
    IFS=$'\t' read -r C5 C7 C30 \
                   T5_IN T5_OUT T5_CC T5_CR \
                   T7_IN T7_OUT T7_CC T7_CR \
                   T30_IN T30_OUT T30_CC T30_CR <<<"$FIRST_LINE"
    IFS=$OLD_IFS

    # Validate: all 15 fields must be non-negative integers
    for v in "$C5" "$C7" "$C30" \
             "$T5_IN" "$T5_OUT" "$T5_CC" "$T5_CR" \
             "$T7_IN" "$T7_OUT" "$T7_CC" "$T7_CR" \
             "$T30_IN" "$T30_OUT" "$T30_CC" "$T30_CR"; do
      if ! [[ "$v" =~ ^[0-9]+$ ]]; then
        echo "CC ?"
        echo "---"
        echo "Malformed TSV row from jq pipeline"
        echo "Last refresh: $NOW_HMS"
        echo "Refresh | refresh=true"
        exit 0
      fi
    done
    ```
    The 15-field numeric regex check catches schema drift: if a new
    `usage` field appears and breaks the column order, validation
    fails and we route to `CC ?`. (opencode LONGTERM Bug 2 mitigation.)

12. **L82-L90 — model list parse** (separate IFS= read for the `M\tmodel\tcount`
    suffix lines):
    ```bash
    MODELS_USED=""
    while IFS=$'\t' read -r tag model count; do
      [[ "$tag" == "M" ]] || continue
      if [[ -z "$MODELS_USED" ]]; then
        MODELS_USED="$model ($count)"
      else
        MODELS_USED="$MODELS_USED, $model ($count)"
      fi
    done < <(echo "$TSV" | tail -n +2)
    ```
    `$MODELS_USED` is empty if no model lines (no usage in any window),
    which is a fine state for the dropdown.

### Section 4: Format menubar + dropdown (L92-L180)

13. **L92-L96 — pct / color helpers** (carried over from opencode
    L28-L43 verbatim — same color thresholds):
    ```bash
    pct_of() {
      awk -v c="$1" -v cap="$2" 'BEGIN{ if (c+0==0) {print "0"} else {printf "%d", (c/cap)*100} }'
    }
    color_for() {
      local pct="$1"
      if [[ "$pct" == "?" ]]; then echo "#ff8800"; return; fi
      if (( pct >= 80 )); then echo "#ff3333"
      elif (( pct >= 50 )); then echo "#ffaa00"
      else echo "#33dd33"
      fi
    }
    ```
    `pct_of` uses awk to avoid bash `$((…))` overflow (opencode R16
    lesson; messages for a 5h window max at 900, well under 2^31,
    so technically bash could handle it, but the habit is correct).

14. **L98-L106 — compute pcts** (5h is the binding constraint, gets
    a real cap; 7d and 30d get treated as informational since caps
    are not publicly documented):
    ```bash
    PCT_5H=$(pct_of "$C5" "$CAP_5H")
    PCT_7D=$(pct_of "$C7" 0)   # 0 cap → "0%" by design; we'll show raw count
    PCT_30D=$(pct_of "$C30" 0)
    C5_COLOR=$(color_for "$PCT_5H")
    ```
    When cap=0, `pct_of` returns "0", which is a misleading number
    for "info-only" windows. The dropdown will show them with
    ` 0% (no published cap)` styling instead. Color is still
    green by default (no cap to exceed).

15. **L108-L120 — menubar line** (target shape `CC 12/45 27%`,
    capped at 14 chars via `length=14`):
    ```bash
    if [[ "$C5" -eq 0 ]]; then
      echo "CC 0 | length=14 tooltip=No Claude usage in last 5h"
    else
      echo "CC ${C5}/${CAP_5H} ${PCT_5H}% | length=14 color=$C5_COLOR tooltip=Claude ${CLAUDE_PLAN}: ${C5} of ${CAP_5H} messages in last 5h"
    fi
    ```
    The `0` case is its own line (mirrors opencode's `OC 0` state
    for "DB exists, no usage today"). 14-char cap is a hard
    SwiftBar convention: the menubar truncates anyway, but the
    cap makes the intent explicit and is used by other plugins
    in the user's setup.

16. **L122 — separator**: `echo "---"`.

17. **L124-L180 — dropdown body** (lines after `---`):
    - `echo "Plan: ${CLAUDE_PLAN} (5h cap: ${CAP_5H}) | color=#888888"`
    - `echo "5h rolling:    ${C5} msg / ${CAP_5H} cap (${PCT_5H}%) | color=$C5_COLOR"`
    - `echo "    resets: server-side, no client signal | color=#888888"`
    - `echo "Weekly:        ${C7} msg (informational) | color=#33dd33"`
    - `echo "Monthly:       ${C30} msg (informational) | color=#33dd33"`
    - `echo "---"`
    - Token totals in 5h window (numfmt for SI suffixes):
      ```bash
      T5_TOTAL=$(awk -v a="$T5_IN" -v b="$T5_OUT" -v c="$T5_CC" -v d="$T5_CR" 'BEGIN{print a+b+c+d}')
      echo "Tokens 5h in:     $(echo "$T5_IN" | "$NUMFMT" --to=si --format="%.1f")"
      echo "Tokens 5h out:    $(echo "$T5_OUT" | "$NUMFMT" --to=si --format="%.1f")"
      echo "Tokens 5h cache:  $(echo "$T5_CC" | "$NUMFMT" --to=si --format="%.1f") created + $(echo "$T5_CR" | "$NUMFMT" --to=si --format="%.1f") read"
      echo "Tokens 5h total:  $(echo "$T5_TOTAL" | "$NUMFMT" --to=si --format="%.1f")"
      ```
    - `echo "Models 5h:       ${MODELS_USED:-none} | color=#888888"`
    - `echo "---"`
    - `echo "Last refresh: $NOW_HMS"`
    - `echo "Refresh | refresh=true"`

## Risks + mitigations

- **C1 — Subscription tier detection is impossible locally.** No file,
  no env var, no DB row reveals whether the user is Pro, Max 5x, or
  Max 20x. Anthropic's server is the only source. *Mitigation*: hardcode
  the 3 documented tiers, env var `CLAUDE_PLAN` to select, default to
  Pro (most common). Document in dropdown which plan is assumed.

- **C2 — Cache_creation/cache_read semantics for message counting.**
  Pro/Max messages are counted per assistant turn. The plan counts
  `type=="assistant"` lines, one per turn. cache_creation / cache_read
  are token-usage sub-fields, not messages — they do NOT add to the
  count. *Mitigation*: count messages, not tokens. Tokens are surfaced
  in the dropdown but not divided by anything that would inflate the
  message count. If Anthropic later switches to a token-based cap,
  the dropdown already has the data; only the cap/percent math changes.

- **C3 — Subagent double-counting.** Subagents are spawned by the
  parent assistant turn and may write to `subagents/agent-*.jsonl`
  with their own `type=="assistant"` lines. Counting them would
  inflate the message total. *Mitigation*: Step 8 explicitly excludes
  `*/subagents/*` from the file list. Document this in dropdown:
  "Excludes subagent messages (counted by parent)."

- **C4 — Many JSONL files → slow parse.** 194 top-level files /
  524MB / ~10-30s jq parse per 5m refresh. *Mitigation*: 5m cadence
  hides the latency (SwiftBar runs in a child process, menubar
  doesn't freeze — the icon just updates 30s late). If user reports
  slowness, v2 adds a tmpfile cache with mtime gate (deferred). If
  parse takes >120s, SwiftBar's process model would still work
  (children are independent) but UX degrades — v2 hard cap at 60s
  via `timeout 60` wrapper if needed.

- **C5 — Schema drift in JSONL.** Anthropic may add fields to `usage`
  (e.g. `cache_creation.ephemeral_1h_input_tokens` already exists in
  the sample), change `type` values, or rename top-level keys.
  *Mitigation*: Step 10's two-stage jq uses `// default` for every
  field (forward-compat for new fields, defensive against missing
  fields). Step 11's 15-field numeric regex catches column-order
  drift. If a new mandatory field appears that breaks the pipeline,
  we route to `CC ?` and the user knows to look at it.

- **C6 — ISO 8601 timestamp parsing in awk mktime.** awk's `mktime`
  on macOS / BSD awk handles `YYYY MM DD HH MM SS` (6 fields). The
  plan's `gsub(/-|T|:|Z|\..*/, " ", ts)` produces exactly that shape.
  *Mitigation*: if a malformed timestamp hits mktime, it returns -1;
  the plan defaults to 0 (epoch 1970), which falls outside every
  window and is naturally excluded. No special handling needed.

- **C7 — Project path encoding.** `~/.claude/projects/-Users-gon-Desktop-foo/`
  uses dash-encoding (slashes → dashes, dots → dashes). The implementer
  must NOT mangle these — pass them verbatim to jq. *Mitigation*:
  Step 8 uses `find ... -print0` + `mapfile -d ''` which preserves
  the path byte-for-byte. No `tr` or `sed` on paths.

- **C8 — bash 3.2.57 IFS leak.** `IFS=$'\t' read` mutates IFS for
  the rest of the script in bash 3.2.57 (no `local` scoping in
  `var= cmd` for `read`). *Mitigation*: every `IFS=$'\t' read` is
  wrapped in `OLD_IFS=$IFS; ...; IFS=$OLD_IFS` (opencode R3 lesson,
  step 11 verbatim).

- **C9 — bash 3.2.57 ERE does not interpret `\t`.** Same as opencode
  R3 — a regex like `^[0-9]+\t[0-9]+$` matches the literal 2-char
  sequence `\t`, not a tab. *Mitigation*: TSV parsing uses
  `IFS=$'\t' read`, never a regex. The numeric validation regex
  `^[0-9]+$` is safe (no `\t` in the pattern).

- **C10 — jq `select(.type=="assistant")` returns 0 on no-match
  (silent zero).** Carried over from opencode LONGTERM Bug 2.
  *Mitigation*: the plan's pipeline produces one output record per
  assistant message; if a file has no assistant messages, the
  pipeline produces no records for that file but the rest of the
  files still flow. The "no data" guard at L75 catches total
  silence. If a schema rename makes `select(.type=="assistant")`
  match nothing globally, `$TSV` is empty and we route to `CC ?`
  via L73-L78.

- **C11 — `set -e` + piped failures.** `for f in ...; do jq ...; done |
  awk ...` — if jq fails on file 1, `set -e` would normally abort
  the script. *Mitigation*: wrap each jq call in `2>/dev/null` and
  catch with `|| true` per the opencode pattern. The `TSV=$(... ||
  TSV="")` outer guard at L70 also makes the whole pipeline
  failure-safe.

- **C12 — 7d / 30d caps are not publicly documented.** Anthropic
  markets Pro/Max as "unlimited" with throttling at the 5h window;
  7d and 30d are not message-capped. *Mitigation*: Step 14 passes
  cap=0 to `pct_of` (which returns "0%"); the dropdown labels
  them as "informational" with raw counts, not %. The user
  understands these are usage trends, not quota burn-downs.

- **C13 — `find -mtime -31` includes the 31st day as a boundary.**
  `-mtime -31` means "modified less than 31 × 24h ago", which is
  actually a 31-day window with the 31st day excluded by 1 second.
  This is fine — the 30d cap is "now - 30d" inclusive, and a
  31-day read just means we have a 1-day buffer for any clock skew.
  *Mitigation*: documented, not changed.

- **C14 — Dropdown `MODELS_USED` may be very long.** A user with
  many Claude models could produce `Models 5h: claude-sonnet-4-5
  (12), claude-opus-4-1 (3), claude-haiku-4-5 (88), ...`. *Mitigation*:
  truncate to first 80 chars with `...` if longer. v2 should show
  a submenu. Deferred.

- **C15 — SwiftBar plugin filename MUST have refresh suffix.**
  *Mitigation*: hardcode `claude.5m.sh`. EC3 enforces.

- **C16 — 5h window is server-side rolling, not local-time
  anchored.** Anthropic's 5h window resets on the server (e.g.
  from the user's first message of the window). The local 5h
  rolling window is an approximation: it counts messages in the
  last 18000s, which is "good enough" most of the time but
  diverges from server near the reset boundary. *Mitigation*:
  the dropdown labels this as "approximate 5h rolling" so the
  user knows. The actual server window is in the JSONL `timestamp`
  field, which is when the client sent the message — that IS
  the right anchor for a "messages I've sent in the last 5h"
  metric. v2 could read `message.usage.cache_creation.ttl` to
  infer window membership more precisely, but the data isn't
  always there.

- **C17 — Refresh is 5m, not real-time.** If the user burns 40
  messages in 5 minutes, the menubar shows a 5m-old number.
  *Mitigation*: 5m is consistent with opencode plugin (same
  filename suffix). For real-time burn-down, the user can
  `Cmd-R` in SwiftBar to force-refresh.

- **C18 — `~/.claude/projects/` may not exist for a brand-new
  user.** Step 6 catches this with `CC: no data`. EC5 enforces.

- **C19 — Project subdirs with special characters.** The dash-encoding
  (C7) makes this rare, but spaces in project paths survive
  (verified by `ls` showing `-Users-gon-Documents-AI-Generated-Apps`
  with no spaces). *Mitigation*: `find -print0` + `mapfile -d ''`
  preserves paths byte-for-byte. If a future path contains a
  newline (extremely unlikely), the plan still handles it.

- **C20 — No `cache_creation` block in old sessions.** Pre-2025
  sessions may have `usage.cache_creation_input_tokens` as a
  flat int (not a nested object with `ephemeral_5m_input_tokens`).
  *Mitigation*: Step 10 uses `.message.usage.cache_creation_input_tokens`
  directly (the field documented in the user context). If the
  flat int is missing, `// 0` defaults to 0. No `cache_creation`
  sub-object is touched in the parse, so the nested shape is
  forward-compat for new sessions and silent-degrade for old
  ones.

- **C21 — Refresh on macOS is throttled by SwiftBar if the plugin
  takes >5s.** SwiftBar itself doesn't throttle (it spawns a new
  process per refresh), but if multiple plugins all run heavy
  work at the same time, the system slows down. *Mitigation*:
  Step 10's 10-30s parse is OK; if it grows, v2 adds the
  background-cache pattern (deferred).

## Exit criteria (oracle commands or check procedure)

The verifier runs these and reports pass/fail. All **contract gates**
(EC1-EC8) must pass.

### Contract gates (required)

- **EC1 — syntax check**:
  `bash -n ~/SwiftBar/claude.5m.sh` exits 0.

- **EC2 — executable bit**:
  `test -x ~/SwiftBar/claude.5m.sh` exits 0.

- **EC3 — filename suffix** (SwiftBar requirement):
  `[[ "$(basename ~/SwiftBar/claude.5m.sh)" =~ \.5m\.sh$ ]]` exits 0.

- **EC4 — metadata present** (SwiftBar requires at least title):
  `grep -q '<xbar.title>' ~/SwiftBar/claude.5m.sh` exits 0.

- **EC5 — graceful no-projects-dir path**:
  `CLAUDE_PROJECTS=/nonexistent/path ~/SwiftBar/claude.5m.sh` exits 0
  AND first stdout line equals `CC: no data`.

- **EC6 — graceful no-jq path** (NEW for Claude; not in opencode):
  `JQ=/nonexistent/jq CLAUDE_PROJECTS=/tmp ~/SwiftBar/claude.5m.sh`
  exits 0 AND first stdout line equals `CC ?`. Use `/tmp` (which
  exists) for `CLAUDE_PROJECTS` to force the jq-missing branch to
  be the one exercised.

- **EC7 — happy path with real data** (skip if
  `~/.claude/projects/` has no `*.jsonl` outside `subagents/`):
  `~/SwiftBar/claude.5m.sh` exits 0 AND first line, AFTER stripping
  SwiftBar params (the `| length=… color=… tooltip=…` suffix),
  matches `^CC ([0-9]+/45|[0-9]+/225|[0-9]+/900|0)$` (default
  Pro plan, cap 45; with `CLAUDE_PLAN=max5x` or `max20x` the
  cap changes).

- **EC8 — synthetic JSONL test** (the analog of opencode EC6c):
  Generate a temp JSONL with N=10 assistant messages in the last
  5h, run the script with `CLAUDE_PROJECTS` pointed at the temp
  dir, assert first line equals `CC 10/45 22%` (with Pro plan).
  Implementation sketch:
  ```bash
  TMPDIR=$(mktemp -d)
  mkdir -p "$TMPDIR/-test-project/aaaa-bbbb-cccc-dddd"
  NOW=$(date -u +%Y-%m-%dT%H:%M:%S.000Z)
  for i in $(seq 1 10); do
    TS=$(date -u -v "-${i}M" +%Y-%m-%dT%H:%M:%S.000Z 2>/dev/null || \
         date -u -d "${i} minutes ago" +%Y-%m-%dT%H:%M:%S.000Z)
    echo "{\"type\":\"assistant\",\"timestamp\":\"$TS\",\"message\":{\"model\":\"claude-sonnet-4-5\",\"usage\":{\"input_tokens\":10,\"output_tokens\":20,\"cache_creation_input_tokens\":0,\"cache_read_input_tokens\":0}}}" >> "$TMPDIR/-test-project/aaaa-bbbb-cccc-dddd/session.jsonl"
  done
  first=$(CLAUDE_PROJECTS="$TMPDIR" ~/SwiftBar/claude.5m.sh | head -1 | sed 's/ *|.*//')
  # Expect: "CC 10/45 22%"
  ```
  The `-v "-${i}M"` (BSD) and `-d` (GNU) dance is needed because
  macOS `date` doesn't have `-d`. The `||` fallback lets the test
  run on Linux too. With 10 messages and Pro cap=45, 22% is the
  expected output.

- **EC8b — synthetic JSONL, zero messages** (also covered by EC7
  via "no data" guards): same setup as EC8 but with a JSONL
  containing only `type=="user"` messages. First line should be
  `CC 0` (we don't count user messages).

- **EC8c — synthetic JSONL, older than 5h**: messages with
  `timestamp` set to 6h ago. First line should be `CC 0` (5h
  window excludes them; 7d and 30d windows include them but
  the menubar is keyed on 5h).

- **EC8d — synthetic JSONL, schema-drift** (forward-compat
  check): messages with `usage` field MISSING entirely. The
  jq pipeline should still produce output (each field defaults
  to 0 via `// default`), so counts should be valid. Assert
  exit 0 and first line is `CC N/45 X%` (not `CC ?`).

- **EC9 — tier override works**: with `CLAUDE_PLAN=max5x`,
  synthetic 50 messages should yield `CC 50/225 22%` (not
  `CC 50/45 111%`). EC9 verifies the cap lookup table.

- **EC10 — aggregate vs per-project** (new mode): with 2
  synthetic projects each with 5 messages, `CLAUDE_PROJECT=`
  (default) yields `CC 10/45`; `CLAUDE_PROJECT=project-a` yields
  `CC 5/45`.

## Considered & rejected

- **Web scrape `claude.ai/settings/usage`.** No public usage API;
  page requires login + CSRF. Auth-token scraping is fragile and
  may violate Anthropic TOS. Rejected.
- **Use `claude-usage-line` CLI as a wrapper.** Verified to exist
  (statusline format), but its output is terminal-style text, not
  JSON, and it doesn't expose raw counts in a parseable form.
  Wrapping a parser around it would be more code than parsing
  the JSONLs directly. Rejected.
- **Use Anthropic API admin endpoint.** Not available to Pro/Max
  consumers; only available to enterprise admins. Rejected.
- **Per-project only (no aggregate).** Loses the "how am I doing
  on my subscription as a whole" view. The user's actual question
  is aggregate; per-project is a secondary feature. Rejected as
  default; accepted as opt-in via `CLAUDE_PROJECT`.
- **1-minute refresh (`.1m.sh`).** Same as opencode — Pro/Max
  5h window is the binding constraint, 1m polling is wasteful
  (4× the JSONL parses per refresh cycle). 5m is plenty.
- **Token-based cap (substitute for message-based).** Not what
  Pro/Max advertises; would surprise the user. Stay message-based
  for the menubar, tokens-in-dropdown as secondary. Rejected as
  the primary metric.
- **Background-cache pattern (parse once, cache, mtime gate).**
  Same as opencode: adds 15-20 lines and a failure mode (stale
  cache). 5m refresh is gentle enough that 10-30s parse is
  acceptable. Defer to v2 if user reports slowness.
- **Show $ in the menubar.** Pro/Max is flat-fee; $ would be
  misleading. Same lesson as opencode R11. Rejected.
- **Show quota RESET time.** Server-side window reset is not
  signaled in JSONLs. Cannot be detected locally. The dropdown
  says "resets: server-side, no client signal" instead of
  fabricating a number. Rejected as fabricated.
- **Parse subagent JSONLs.** Would double-count messages. Default
  to exclude; document the choice. Rejected as default; could
  be opt-in via `CLAUDE_INCLUDE_SUBAGENTS=1` in v2.
- **Read all JSONLs (no mtime filter).** 2797 files, 1GB+. With
  the 30d window, only 194 are relevant. mtime filter cuts
  read time by ~14×. Rejected — mtime filter is the right
  default.
- **Use Python for JSON parsing instead of jq.** `python3 -c` is
  reliable but adds a dependency and ~50ms startup. jq is
  already on PATH and is what opencode uses. Stay consistent
  with opencode. Rejected.
- **Streaming parser (incremental jq).** Same as background-cache:
  not needed for 5m cadence. Defer to v2.
- **Cross-validate against a server-side fetch.** No public API.
  Local is the only source. Rejected (impossible).
- **Build a sibling "tier detector" plugin that scrapes
  claude.ai/usage once a day.** Same problem as the web scrape
  above. Rejected.

## Notes for implementer

- **The pipeline is `find → jq → jq → awk`.** Test it as a single
  command from a shell before pasting into the script. Sample:
  ```bash
  echo '{"type":"assistant","timestamp":"2026-06-18T18:00:00.000Z","message":{"model":"claude-sonnet-4-5","usage":{"input_tokens":10,"output_tokens":20,"cache_creation_input_tokens":0,"cache_read_input_tokens":0}}}' \
    | jq -c 'select(.type == "assistant") | {ts: .timestamp, in: .message.usage.input_tokens, out: .message.usage.output_tokens, cache_create: .message.usage.cache_creation_input_tokens, cache_read: .message.usage.cache_read_input_tokens, model: .message.model}' \
    | jq -r '"\(.ts // "1970-01-01T00:00:00.000Z")|\(.in // 0)|\(.out // 0)|\(.cache_create // 0)|\(.cache_read // 0)|\(.model // "?")"'
  ```
  Expected output: `2026-06-18T18:00:00.000Z|10|20|0|0|claude-sonnet-4-5`.

- **Use `IFS=$'\t' read`, NOT a regex, for TSV parsing.** opencode
  R3 / C9 lesson — macOS bash 3.2.57 ERE does not interpret `\t`.
  A regex would reject every real row. Wrap in `OLD_IFS=$IFS;
  IFS=$'\t' read -r ... <<<"$ROW"; IFS=$OLD_IFS` to prevent
  IFS leak.

- **The 15-field numeric validation regex `^[0-9]+$` is safe**
  (no `\t` in pattern). Apply it to every variable after the
  read. This catches schema drift that breaks column order.

- **Default plan is `pro` (45 msg/5h cap).** User can override
  with `CLAUDE_PLAN=max5x` (225) or `CLAUDE_PLAN=max20x` (900).
  Other values fall back to Pro with a quiet default. The
  dropdown labels the assumed plan so the user can spot a
  wrong assumption.

- **The 7d and 30d windows are INFORMATIONAL.** No public caps.
  Show raw counts, label as "informational", don't try to
  compute %. The 5h window is the only binding constraint.

- **Skip `*/subagents/*` JSONLs by default.** Subagent messages
  are spawned by the parent assistant turn and would double-count.
  Use `find -not -path "*/subagents/*"`. v2 may add a
  `CLAUDE_INCLUDE_SUBAGENTS=1` toggle.

- **Test all 4 distinct empty/failure paths** before declaring
  done:
  1. `CLAUDE_PROJECTS=/nonexistent ./claude.5m.sh` → `CC: no data`
  2. `JQ=/nonexistent/jq ./claude.5m.sh` → `CC ?` (jq missing)
  3. `CLAUDE_PROJECTS=<empty-dir> ./claude.5m.sh` → `CC 0`
     (dir exists, no JSONL files)
  4. Synthetic JSONL with 10 assistant messages → `CC 10/45 22%`
     (happy path, EC8)

- **Token arithmetic uses awk, not bash `$((…))`.** awk coerces
  strings to numbers; bash `$((…))` overflows at 2^63. Cache
  totals can reach 10^9 in heavy sessions.

- **No code comments** beyond the SwiftBar metadata header in
  L2-L6. The `set -euo pipefail` line, var assignments, jq, and
  echo calls stay unannotated.

- **The `length=14` cap is for the menubar line only.** It
  doesn't truncate the dropdown. SwiftBar's menubar auto-truncates
  anyway, but the cap is the convention used in opencode and
  other plugins in the user's setup.

- **SwiftBar `color=` parameter is per-line.** Each dropdown line
  can have its own color. Use `#33dd33` (green), `#ffaa00` (yellow),
  `#ff3333` (red), `#ff8800` (orange for "?" state), `#888888`
  (gray for informational).
