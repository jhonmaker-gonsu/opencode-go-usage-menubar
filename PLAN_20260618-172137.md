# Plan: SwiftBar plugin for OpenCode Go usage

## Goal

A single executable SwiftBar plugin that surfaces **today's** OpenCode usage
(tokens + estimated cost) in the macOS menubar, wrapped around the
`opencode-usage` CLI from `gaboe/opencode-usage` on npm.

Final deliverable:

- `~/SwiftBar/opencode.5m.sh` — the plugin itself.
- This plan file (workdir scratch only).

Menubar line shape (target): `OC 16.1K` (tokens only — cost is API-equivalent
on a flat-fee Go subscription, showing a dollar number in the menubar is
misleading per the npm README and the user's actual question
"How much have I used already?").
Dropdown shape (target): date label, in/out/total tokens, cost (labelled
"API-equivalent; Go is flat-fee"), Go limits hint, refresh action. Menubar
line is capped to 14 chars via `length=14`. **No dashboard link in v1**
(deferred — SwiftBar's `terminal=`/`href=` syntax needs verification
on this SwiftBar version, and the Commander web dashboard port is
user-configurable in `opencode-usage`).

## Constraints

- **Stake**: small (single bash script, ~30–50 lines including metadata + dropdown).
- **Output file is the SwiftBar plugin** — SwiftBar mandates specific header
  comments (`<xbar.title>`, `<xbar.dependencies>`, etc.). These are
  tool-required metadata, not AI commentary; the no-comments rule is
  relaxed for them.
- **No emojis** in code or plan (user said no).
- **MD for the plan file**; shell for the script.
- **No global install** of `opencode-usage` assumed; use `bunx` (zero-install).
- **Tooling already present** on this Mac: `bun` (`/Users/gon/.bun/bin/bun`),
  `jq` (`/opt/homebrew/bin/jq`), `numfmt` (`/opt/homebrew/bin/numfmt`).
  SwiftBar is NOT installed — the script must still work standalone
  (runnable from a terminal to test).
- **Env is macOS** — BSD `date` + GNU `numfmt` from Homebrew. No `gdate`.
- **Only today** in the menubar line; richer detail in the dropdown.
- **Multi-agent workflow** — planner (this file) → implementer → verifier.
  Verifier will run mechanical oracle commands; they must be specified here.

## Approach (numbered, file:line targeted)

Target file: `~/SwiftBar/opencode.5m.sh` (single file, ~30–50 lines).
Line numbers are approximate; the implementer may re-flow within a band.

1. **L1 — shebang**: `#!/usr/bin/env bash`.
2. **L2–L6 — SwiftBar metadata header** (required, not commentary):
   - `# <xbar.title>OpenCode Go Usage</xbar.title>`
   - `# <xbar.version>1.0.0</xbar.version>`
   - `# <xbar.author>gon</xbar.author>`
   - `# <xbar.desc>Today's OpenCode token count (opencode-usage)</xbar.desc>`
   - `# <xbar.dependencies>bun,jq,numfmt</xbar.dependencies>`
   - (dropped `<swiftbar.refreshOnOpen>` — 5m filename refresh is the
     canonical mechanism; the `<swiftbar.*>` metadata namespace for
     plugin refresh is not the standard path)
3. **L8 — strict mode**: `set -euo pipefail`. (We deliberately want
   non-zero exit on bunx failure so the `?` fallback can take over — see
   risk R2; `set -e` is fine because we wrap the bunx call in a guard.)
4. **L9–L13 — tool paths with overrides for testability**:
   - `BUN="${BUN:-/Users/gon/.bun/bin/bun}"`
   - `JQ="${JQ:-/opt/homebrew/bin/jq}"`
   - `NUMFMT="${NUMFMT:-/opt/homebrew/bin/numfmt}"`
   - `OC_DB="${OC_DB:-$HOME/.local/share/opencode/opencode.db}"`
5. **L14–L18 — DB existence guard** (graceful no-data path):
   - If `[[ ! -f "$OC_DB" ]]`, print `OC: no data` and exit 0.
6. **L19–L24a — fetch JSON via bunx, suppressed, with explicit failure branch**:
   - **Must use `--stats --json`** — `--json` alone launches the
     interactive TUI and hangs. Verified empirically: the CLI's `--help`
     shows `--json` requires a non-default mode flag.
   - Capture exit code BEFORE the if-test (bash resets `$?` to 0 inside
     `if` body — using `$?` inside the body would print "exit 0" which
     is misleading):
     ```bash
     JSON=$("$BUN" bunx --bun opencode-usage --stats --json --since 1d 2>/dev/null)
     RC=$?
     if [[ $RC -ne 0 ]]; then
       echo "OC ?"
       echo "---"
       echo "opencode-usage CLI failed (exit $RC)"
       echo "Try: bunx --bun opencode-usage --stats --json --since 1d"
       echo "Refresh | refresh=true"
       exit 0
     fi
     ```
   - `2>/dev/null` covers bunx cache noise ("Resolving dependencies…",
     "Saved lockfile") on every call.
7. **L25–L31 — jq parse today's row (with RC capture)**:
   - **Schema verified** by running the CLI: top-level object with
     `periods` array; each period has `date`, `models`, `providers[]`,
     and `totals: {input, output, cost}`. The plan's original
     `inputTokens`/`outputTokens`/`totalTokens` selector would have
     matched nothing.
   - **Capture jq exit status** so schema drift (jq returns non-zero)
     can be distinguished from "no usage today" (jq returns zero with
     empty stdout). Without RC capture, schema drift silently
     downgrades to the "OC 0" state — wrong UX.
   - Selector:
     ```bash
     TODAY=$(date +%Y-%m-%d)
     JQ_RC=0
     if ! ROW=$(echo "$JSON" | "$JQ" -r --arg t "$TODAY" '
       .periods[]? | select(.date == $t)
       | [.totals.input // 0, .totals.output // 0, .totals.cost // 0]
       | @tsv' 2>/dev/null); then
       JQ_RC=$?
       ROW=""
     fi
     ```
   - The `[]?` on `.periods` tolerates the field being missing or
     non-iterable. It does NOT tolerate the entire response being a
     top-level array — that would need a different selector shape;
     for now jq-error on that shape safely routes to `OC ?` via the
     RC branch.
   - `--arg t` is quoted-safe; `$TODAY` is `YYYY-MM-DD` which matches
     the CLI's emitted `.date` format (see R17 for TZ caveat).
8. **L32–L50 — parse row, guard empty / malformed / schema-drift**:
   - **DO NOT use a regex on the row.** macOS bash 3.2.57's ERE does
     not treat `\t` as a tab; a regex like `^[0-9]+\t[0-9]+\t[0-9]+$`
     matches the literal two-char sequence `\t`, so it would reject
     every real row. Use `IFS=$'\t' read` instead. **Restore IFS
     afterward** to avoid leaking the tab-separator into later
     word-splitting in bash 3.2.57 (no `local` scope for `var= cmd`).
   ```bash
   if [[ $JQ_RC -ne 0 ]]; then
     echo "OC ?"
     echo "---"
     echo "jq parse failed (exit $JQ_RC) — opencode-usage schema may have changed"
     echo "Refresh | refresh=true"
     exit 0
   fi
   if [[ -z "$ROW" ]]; then
     echo "OC 0 | length=14 tooltip=No usage today"
     echo "---"
     echo "Date: $TODAY"
     echo "Tokens in: 0"
     echo "Tokens out: 0"
     echo "Total: 0"
     echo "Cost est.: \$0.00"
     echo "(API-equivalent; Go is flat-fee) | color=#888888"
     echo "Go limits: 5h=\$12, weekly=\$30, monthly=\$60 | color=#888888"
     echo "---"
     echo "Refresh | refresh=true"
     exit 0
   fi
   OLD_IFS=$IFS
   IFS=$'\t' read -r IN OUT COST_RAW <<<"$ROW"
   IFS=$OLD_IFS
   if [[ -z "$IN" || -z "$OUT" || -z "$COST_RAW" ]]; then
     echo "OC ?"
     echo "---"
     echo "Malformed row from opencode-usage (3 fields expected: in, out, cost)"
     echo "Refresh | refresh=true"
     exit 0
   fi
   ```
   - This routes four distinct conditions to four distinct menubar
     lines: `OC: no data` (no DB, L14-L18), `OC ?` (CLI failure,
     L19-L24a), `OC ?` (jq schema-drift, L32-L37), `OC 0` (no usage
     today, L38-L50), `OC ?` (malformed fields, L51-L57),
     `OC <num>[kmgt]?` (happy path, L58+).
9. **L35–L42 — format menubar line (tokens only, capped length)**:
   - `TOTAL_TOKENS=$(awk -v a="$IN" -v b="$OUT" 'BEGIN{print a+b}')`
     (use awk to avoid bash `$((…))` overflow on 2^63 — IN+OUT for
     heavy days can exceed int32).
   - `TOKENS_SI=$(echo "$TOTAL_TOKENS" | "$NUMFMT" --to=si --format="%.1f")`
     → `16.1k` (GNU numfmt outputs lowercase suffix).
   - `echo "OC ${TOKENS_SI} | length=14 tooltip=OpenCode Go usage today"`
     (cost is intentionally NOT in the menubar; it's in the dropdown
     behind an "API-equivalent" label).
10. **L43 — separator**: `echo "---"`.
11. **L44–L55 — dropdown body** (lines after `---`):
    - `echo "Date: $TODAY"`
    - `echo "Tokens in: $(echo "$IN" | "$NUMFMT" --to=si --format="%.1f")"`
    - `echo "Tokens out: $(echo "$OUT" | "$NUMFMT" --to=si --format="%.1f")"`
    - `echo "Total: ${TOKENS_SI}"`
    - `COST_FMT=$(printf '$%.4f' "$COST_RAW" 2>/dev/null | sed 's/0\+$//; s/\.$//')`
      (4 decimal places then strip trailing zeros; `0.0042` → `$0.0042`,
      `0.10` → `$0.10`, `1.0` → `$1`).
    - `echo "Total cost (API-equivalent): ${COST_FMT}"`
    - `echo "(Go subscription is flat-fee) | color=#888888"`
    - `echo "Go limits: 5h=\$12, weekly=\$30, monthly=\$60 | color=#888888"`
    - `echo "---"`
    - `echo "Refresh | refresh=true"`
    - **Drop** the original "Run opencode-usage --commander" menu item
      (SwiftBar's `params=` syntax was malformed; defer to follow-up).
12. **L56 — `chmod +x`** done by SwiftBar at install time per docs, but
    the install step (Step 14 below) also runs it explicitly to be safe.

Install step (run by hand once, not part of the plugin):

13. `mkdir -p ~/SwiftBar` (idempotent; user said this dir was just created).
14. Drop `opencode.5m.sh` into `~/SwiftBar/`, `chmod +x`.
15. Optional: open SwiftBar, point it at `~/SwiftBar`.

## Risks + mitigations

- **R1 — `opencode-usage` not installed locally and `bunx` first-run is slow.**
  bunx downloads the package on first call; can take 10–30s and stalls
  the menubar. *Mitigation*: this is one-time, `bunx` caches to
  `~/.bun/install/cache/`, subsequent runs are 0.5–2s. 5-minute refresh
  hides the cost. Document that the first refresh will be slow.
  If the cache is corrupted (e.g. `FileNotFound: node_modules`),
  removing `/private/var/folders/*/T/bunx-*` and retrying fixes it.
- **R2 — `opencode-usage --json` alone launches the interactive TUI.**
  Verified: the CLI's default mode is interactive; `--json` requires
  the `--stats` mode flag. *Mitigation*: Step 6 explicitly uses
  `--stats --json --since 1d`. The first run of `--stats` may take
  longer than `--json` alone (it queries the DB); that's fine.
- **R3 — `~/.local/share/opencode/opencode.db` doesn't exist yet
  (user hasn't run OpenCode today, or v1.1.x without SQLite).**
  *Mitigation*: L14–L18 existence guard prints `OC: no data` and exits 0.
  Testable via `OC_DB=/nonexistent ...`.
- **R4 — JSON schema of `opencode-usage --json` is undocumented but verified.**
  R2 confirmed by running the CLI:
  `{periods:[{date,models,providers:[{id,input,output,cacheRead,cost}], totals:{input,output,cost}}], totals:{input,output,cost}}`.
  Selector is locked to that shape with `[]?` for forward-compat.
- **R5 — SwiftBar not installed on this Mac.**
  *Mitigation*: the script is a plain bash script — `bash opencode.5m.sh`
  prints the menubar line to stdout and can be eyeballed in the
  terminal. SwiftBar's absence is orthogonal to script correctness.
- **R6 — macOS BSD `date` vs GNU `date`.**
  *Mitigation*: only use `date +%Y-%m-%d` — identical output on both.
  Don't use `date -d` or `date -u` (would diverge).
- **R7 — `bunx` not in PATH, only `/Users/gon/.bun/bin/bun` is.**
  *Mitigation*: never call `bunx` directly; always call
  `"$BUN" bunx --bun ...` (Step 6). `bunx` is a built-in subcommand of bun.
- **R8 — SwiftBar stderr spam from bunx install noise.**
  *Mitigation*: `2>/dev/null` on the bunx call. Also covers the CLI's
  own progress messages on the first `--stats` invocation.
- **R9 — Menubar freeze if bunx takes >5s.**
  *Mitigation*: 5-minute refresh (not 1-minute); SwiftBar runs the
  plugin in a child process so the menubar animation itself doesn't
  freeze, only the icon update is delayed. Acceptable for a status
  indicator. SwiftBar has no plugin-timeout knob; if freezes are
  reported, add a background-cache pattern with `nohup … &` and an
  mtime gate (out of scope for v1).
- **R10 — SwiftBar plugin filename MUST have refresh suffix.**
  *Mitigation*: hardcode `opencode.5m.sh`. Exit criterion EC3 enforces.
- **R11 — Cost is API-equivalent, not subscription cost. CONFIRMED via npm README.**
  On a flat-fee Go subscription ($5/$10), a `$0.42` menubar number
  is actively misleading. *Mitigation* (hard): menubar line shows
  tokens only (`OC 16.1K`); cost appears in the dropdown with the
  explicit label `(API-equivalent; Go is flat-fee)` so the user is
  never surprised.
- **R12 — jq selector returns malformed output (schema drift, partial data).**
  *Mitigation*: L32–L50 empty-string-validates the parsed fields
  (`IN`, `OUT`, `COST_RAW`) before downstream use. A missing field
  routes to the `OC ?` fallback. A jq non-zero exit is now
  distinguished from empty-stdout via the `JQ_RC` capture and
  surfaced with a different error message.
- **R13 — Plugin header tag set is wrong.**
  Plan uses `<xbar.*>` (xbar.app convention) AND `<swiftbar.*>` for
  `refreshOnOpen`. SwiftBar accepts both — verified via the SwiftBar
  README which documents both prefixes as supported. EC4 enforces
  the presence of `<xbar.title>` (the most widely-supported form).
- **R14 — `cost` field type may be a string, not a number.**
  Many CLIs serialize money as strings (e.g. `"0.0042"`). If
  `opencode-usage` does this, the IFS+read approach still parses it
  (read doesn't care about type), but downstream `awk` math could
  fail. *Mitigation*: wrap arithmetic in `awk` (it does
  string-to-number coercion). EC8 is extended to assert the type.
- **R15 — Three distinct empty/failure paths were conflated as `OC ?`.**
  Original plan printed `OC ?` for (a) bunx CLI failure, (b) jq error,
  (c) DB missing, (d) no usage today. From a user's perspective these
  are different states. *Mitigation*: differentiate:
  - DB missing → `OC: no data` (L14-L18)
  - No usage today (DB exists, jq found no row) → `OC 0` (L30-L34)
  - bunx CLI failure → `OC ?` (L19-L24a)
  - jq error / malformed row → `OC ?` (L30-L34 second branch)
  This is a UX fix, not a security fix.
- **R16 — bash `$((…))` overflows at 2^63.**
  Token arithmetic in step 9 uses `awk` to avoid this. Heavy days
  (multi-million tokens in/out) would have crashed bash arithmetic.
  Documented for the implementer.
- **R17 — TZ mismatch between local `date` and CLI's `.date` field.**
  `date +%Y-%m-%d` uses local TZ; the CLI may emit UTC dates. Near
  midnight in non-UTC TZs the jq filter may match the wrong calendar
  day or no day at all. *Mitigation*: probe the CLI payload once
  (`... | jq '.periods[0].date'`) to determine TZ. If UTC, set
  `TODAY=$(TZ=UTC date +%Y-%m-%d)`. Bake the chosen TZ into L96
  before declaring done. For users east of UTC the first morning
  invocation is the riskiest; west of UTC the last evening
  invocation is.
- **R18 — `<swiftbar.refreshOnOpen>` is not in SwiftBar's standard
  metadata tag list.** Dropped in this revision. The 5m filename
  refresh (`opencode.5m.sh`) is the canonical mechanism. Removing
  this tag also removes one potential source of confusion in
  SwiftBar's plugin-metadata documentation.
- **R19 — Go plan limits hardcoded in dropdown will go stale.**
  `5h=$12, weekly=$30, monthly=$60` is current as of the plan date
  but not source-cited. *Mitigation*: append a `| color=#888888`
  suffix (already present) and accept the staleness risk for v1. v2
  should source the limits from a single config line at the top of
  the plugin so updating one number is one edit.

## Exit criteria (oracle commands or check procedure)

The verifier runs these and reports pass/fail. All **contract gates**
(EC1–EC6) must pass. **Smoke gates** (EC7–EC8) run only if the upstream
CLI and DB are available; they self-skip cleanly otherwise.

### Contract gates (required)

- **EC1 — syntax check**:
  `bash -n ~/SwiftBar/opencode.5m.sh` exits 0.

- **EC2 — executable bit**:
  `test -x ~/SwiftBar/opencode.5m.sh` exits 0.

- **EC3 — filename suffix** (SwiftBar requirement):
  `[[ "$(basename ~/SwiftBar/opencode.5m.sh)" =~ \.5m\.sh$ ]]` exits 0.

- **EC4 — metadata present** (SwiftBar requires at least title):
  `grep -q '<xbar.title>' ~/SwiftBar/opencode.5m.sh` exits 0.

- **EC5 — graceful no-DB path**:
  `OC_DB=/nonexistent/path/to/db ~/SwiftBar/opencode.5m.sh` exits 0
  AND first stdout line equals `OC: no data`. Assertion:
  `[[ "$(OC_DB=/nonexistent ... | head -1)" == "OC: no data" ]]`.

- **EC6 — graceful no-bunx path (TIGHTENED, real-failure branch)**:
  `OC_DB=/etc/passwd BUN=/nonexistent/bun ~/SwiftBar/opencode.5m.sh`
  exits 0 AND first stdout line equals `OC ?` (literal space +
  question mark, NOT `OC 0` or `OC: no data`). The
  `OC_DB=/etc/passwd` trick forces the L14 DB guard to pass so the
  bunx-failure branch is the one being exercised.

- **EC6b — no-usage-today path (DB exists, zero tokens)**:
  **Dropped** (was weak-gate in iter 1). Replaced by EC6c which
  exercises the same code path with a stronger contract: the
  implementer must run the script against a synthetic DB whose
  `periods` array has the row for today with all-zero totals, and
  assert first line equals `OC 0`. Synthetic DB generation is
  documented in EC6c. If synthetic-DB generation is impractical
  in the verifier env, EC6c self-skips with a clear log message;
  in that case the no-usage-today case is **declared untested** and
  the implementer's eyeball test (Notes for implementer, item 3.3)
  is the only verification.

- **EC6c — synthetic zero-tokens DB path** (preferred over EC6b):
  Generate a temp JSON file shaped like the real CLI output, but
  with a today's row where `.totals = {input:0, output:0, cost:0}`.
  Stub the `opencode-usage` call to read from this file (via a
  `BUN` override that runs `cat` instead of `bunx`). Assert first
  line equals `OC 0`. Implementation:
  ```bash
  cat > /tmp/oc-zero.json <<EOF
  {"periods":[{"date":"$(date +%Y-%m-%d)","models":[],"providers":[],"totals":{"input":0,"output":0,"cost":0}}],"totals":{"input":0,"output":0,"cost":0}}
  EOF
  BUN=/bin/cat first=$(cat /tmp/oc-zero.json | ~/SwiftBar/opencode.5m.sh | head -1 | sed 's/ *|.*//')
  # Note: BUN=/bin/cat means the script's bunx call is replaced by
  # "cat /tmp/oc-zero.json" — but the script's BUN var is treated as
  # the bunx binary path, not as an arbitrary command. This EC is
  # aspirational; if BUN-override is too tightly coupled, the
  # verifier should fall back to EC6b (skip cleanly) and rely on
  # the implementer's eyeball test.
  ```

### Smoke gates (run if env allows, skip cleanly otherwise)

- **EC7 — happy path with real opencode-usage** (skip if
  `~/.local/share/opencode/opencode.db` does not exist):
  `~/SwiftBar/opencode.5m.sh` exits 0 AND first line, AFTER stripping
  SwiftBar params (the `| length=… tooltip=…` suffix), matches
  `^OC [0-9.]+[kmgt]?$` (suffix optional, allows `OC 0`).
  Strip command:
  ```bash
  first=$(~/SwiftBar/opencode.5m.sh | head -1 | sed 's/ *|.*//')
  [[ "$first" =~ ^OC\ [0-9.]+[kmgt]?$ ]]
  ```
  The plan's `length=14 | tooltip=…` SwiftBar params are stripped
  before regex matching.

- **EC8 — JSON schema probe (mandatory if EC7 runs)**:
  `"$BUN" bunx --bun opencode-usage --stats --json --since 1d 2>/dev/null | jq .`
  produces parseable JSON whose top-level has a `.periods` array.
  If this fails the jq selector in the plugin needs revision and
  the plugin MUST be considered broken (do not ship).

- **EC8a — schema shape check**:
  `... | jq '.periods[0] | (.date|type) + "|" + (.totals|type)'`
  should print `string|object` (date is a string, totals is an
  object). Anything else aborts with a clear failure.

- **EC8b — type check on totals**:
  `... | jq '.periods[0].totals | (.input|type) + "|" + (.output|type) + "|" + (.cost|type)'`
  should print `number|number|number`. If `cost` is a string, the
  awk arithmetic needs `tonumber?` coercion (the `IFS= read` is
  type-agnostic; the issue is downstream math).

## Considered & rejected

- **Native Swift menubar app (`NSStatusItem`).** Heavy: requires Xcode,
  code signing, separate distribution. User explicitly framed the task
  as a "SwiftBar plugin". Rejected.
- **Raycast extension.** Out of scope; doesn't live in the menubar
  passively. User said "menubar widget". Rejected.
- **Globally install `opencode-usage` (`bun add -g`)**. Pros: ~100ms
  per call vs ~1–2s for bunx. Cons: requires `npm`/`bun` global path
  management, polluting user env. bunx is recommended upstream and
  zero-install. Rejected for v1; can revisit if R1 manifests.
- **`.1m.sh` instead of `.5m.sh`** (1-minute refresh). Rejected: Go
  quota windows are 5h/weekly/monthly — 1m refresh is wasteful, ~300
  CLI calls/day per plugin. 5m = 60/day, plenty for a status indicator.
- **Streamable plugin type.** Rejected: opencode-usage is
  request/response, not a stream. Standard plugin is correct fit.
- **Reading `opencode.db` directly via `sqlite3`.** Pros: zero CLI
  dependency. Cons: SQLite schema is private to OpenCode, may change
  between versions; bypasses the cost-calculation logic in
  `opencode-usage` (which knows model pricing). Wrapping the CLI is
  the user's explicit ask. Rejected.
- **Filtering to `opencode-go` provider only** via `--provider`. R1
  reviewer suggested defaulting to `--provider opencode`; rejected for
  v1 because (a) the user has not asked to filter, (b) other providers
  the user may have configured in OpenCode still consume from the Go
  wallet only if the model is Go-hosted — `opencode-usage` already
  attributes cost correctly per model, and (c) the action of the
  question "how much have I used" is "how much of the Go plan" — see
  R11; the cost number is API-equivalent either way. Defer to a
  follow-up; trivial one-line tweak (`OC_ARGS=(--provider opencode)`).
- **Background-cache pattern** (write JSON to a tmpfile via
  `nohup … &`, gate by mtime). Rejected for v1: adds 15–20 lines and a
  failure mode (stale cache). 5m refresh is gentle enough that
  occasional 1–2s menubar lag is acceptable. Keep the implementation
  small.
- **Showing cost in the menubar line.** Considered and REJECTED for v1:
  on a flat-fee Go plan the cost number is API-equivalent and
  misleading per R11. Menubar line is tokens-only; cost is in the
  dropdown with the explicit "API-equivalent" label.
- **Showing `5h $X/$12` (remaining quota) in the menubar.** Considered:
  requires daily/weekly/monthly aggregation across providers and
  depends on which Go models the user actually used; a one-line
  number is too lossy. Defer to a follow-up if user wants quota
  visibility.
- **Composing the bash with a helper function (`fetch_usage`).**
  Reconsidered after R1's finding that `|| true` swallows the bunx
  error indistinguishably from a no-data day. Replaced with an
  explicit `if ! JSON=$(...); then ... fi` branch.
- **Commander dashboard menu item.** Original spec had a "Run
  commander" item with `params=commander=--commander,terminal=false`.
  SwiftBar's `params=` is positional, not key=value; the syntax was
  malformed. Rejected — drop the item, defer to follow-up.
- **Open dashboard link (any form).** Iter 1's target shape promised
  one, iter 2's body did not deliver it, and the `terminal=` and
  `href=` SwiftBar actions need verification on this SwiftBar version
  (the Commander web port is user-configurable so `href=` would
  require a stable default or another env var). Rejected for v1 to
  keep the dropdown small and match the existing deferral pattern.
  The opencode web dashboard at `https://opencode.ai/auth` is
  reachable in any browser; the user does not need a one-click
  menubar shortcut.
- **`<swiftbar.refreshOnOpen>` metadata tag.** Rejected: not in
  SwiftBar's standard metadata list. The 5m filename refresh
  (`opencode.5m.sh`) is the canonical mechanism.

## Notes for implementer

- **Use `--stats --json`** — never `--json` alone. Without `--stats`
  the CLI launches the interactive TUI and the plugin will hang
  SwiftBar's refresh worker.
- **jq selector is now locked** to the verified schema (Step 7). Do
  not "improve" it back to the old `inputTokens`/`outputTokens` form.
- **jq assignment MUST be wrapped in `if ! ROW=$(…); then ROW=""; fi`
  AND `JQ_RC=$?` captured** — without RC capture, a schema-drift jq
  failure silently downgrades to the `OC 0` "no usage today" state,
  which is wrong UX.
- **Row parsing uses `IFS=$'\t' read -r IN OUT COST_RAW <<<"$ROW"`,
  NOT a regex.** macOS bash 3.2.57's ERE does not interpret `\t` as a
  tab. A regex would reject every real row.
- **Restore IFS after read** with `OLD_IFS=$IFS; IFS=$'\t' read …;
  IFS=$OLD_IFS`. bash 3.2.57 leaks IFS changes; bash 4+ scopes them
  to the `read` builtin. The leak is benign today (no later
  word-splitting) but a future maintainer adding a `for x in $foo`
  would silently break.
- **Test all three failure paths** before declaring done:
  1. `OC_DB=/nonexistent ./opencode.5m.sh` → first line `OC: no data`
  2. `OC_DB=/etc/passwd BUN=/nonexistent/bun ./opencode.5m.sh` →
     first line `OC ?` (with the question mark)
  3. With real opencode.db, no usage today (e.g. a `~/.local/share/opencode/opencode.db`
     with no rows for `$(date +%Y-%m-%d)`) → first line `OC 0`. If
     you can't easily simulate "no usage today", skip this eyeball
     test and rely on the verifier.
- **Token arithmetic uses awk, not bash `$((…))`**. awk coerces
  strings to numbers; bash `$((…))` overflows at 2^63. Heavy days
  (millions of tokens) would have crashed bash arithmetic.
- **Cost formatting**: 4 decimal places then strip trailing zeros
  (`0.0042` → `$0.0042`, `1.0` → `$1`). 2 decimal places is too coarse
  for sub-cent costs.
- **Do not add code comments** beyond the SwiftBar metadata header
  in L2–L7. The `set -euo pipefail` line, var assignments, jq, and
  `echo` calls stay unannotated.
